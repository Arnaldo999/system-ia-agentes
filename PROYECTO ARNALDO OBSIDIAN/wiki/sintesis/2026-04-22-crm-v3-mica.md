# CRM v3 Mica — replicación v3 en Airtable

## Síntesis ejecutiva

El 22 de abril de 2026, en la misma jornada que se completó el refactor v3 en Robert, se replicó el modelo completo en Mica respetando su stack exclusivo (Airtable + Easypanel + paleta ámbar).

Funcionalmente idéntico a Robert. Técnicamente distinto por las limitaciones de Airtable vs Postgres: no hay transacciones, los FK son `linkedRecord`, los arrays son `multipleSelects`, los IDs son strings `rec...`. Pero desde el punto de vista del usuario inmobiliario, la experiencia es exactamente la misma que Robert:

- Modal unificado 3 pasos para crear contrato (Cliente → Activo → Contrato)
- 3 puertas de entrada (panel Clientes / click lote / convertir lead)
- Ficha de cliente con tab Relaciones mostrando ficha 360
- Sidebar acordeón con 3 grupos GESTIÓN
- Paneles editables Inmuebles/Inquilinos/Pagos/Liquidaciones (antes placeholders)
- Persona única con roles acumulables (comprador/inquilino/propietario)

## Arquitectura adaptada a Airtable

### Schema — 39 campos nuevos distribuidos en 8 tablas

Ninguna tabla eliminada. Sólo se agregaron campos + links para hacer explícitas las relaciones que antes estaban implícitas (strings libres).

**Tabla clave `CLIENTES_ACTIVOS`** ahora tiene:
- `Roles` (multipleSelect: comprador/inquilino/propietario)
- `Lead_Origen` (link → Clientes) — trazabilidad del origen bot
- `Asesor_Asignado` (link → Asesores)
- `Apellido`, `Documento`, `Ciudad`, `Origen_Creacion`

**Tabla `Contratos`** (polimórfico multi-link):
- `Cliente` (link → CLIENTES_ACTIVOS)
- `Lote_Asignado` + `Propiedad_Asignada` + `Inmueble_Asignado` (3 links separados, solo UNO usado por contrato)
- `Item_Descripcion` para guardar subtipo granular (ej "venta_lote - Palo Alto L-5")
- `Cuotas_*`, `Monto_Cuota`, `Proximo_Vencimiento`, `Estado_Pago`

### Backend — `db_airtable.py` +978 líneas

Helpers genéricos `_at_*` + serializers polimórficos + 12 endpoints nuevos espejando Robert. El CRM de Mica hoy responde 11/11 GET de GESTIÓN con 200 OK.

### Frontend — 5 JS nuevos + crm-v2.html +657 líneas

Todo el código portable del frontend Robert se adaptó a Mica en un día. Cambios no portables:
- Paleta purple → ámbar (64 ocurrencias `#f59e0b`, 0 de purple)
- `SAAS_API` → `agentes.arnaldoayalaestratega.cloud` (backend Mica)
- IDs como strings (sin `parseInt`) — IDs Airtable son `rec...`

## Pregunta de origen

¿Cómo replicar en Mica (Airtable) el mismo modelo CRM v3 que se acaba de terminar en Robert (Postgres), manteniendo paridad funcional pero respetando el stack exclusivo de Mica (sin Postgres, con Airtable como única DB)?

## Trade-offs aceptados vs Robert

| Dimensión | Robert | Mica |
|-----------|--------|------|
| **Integridad referencial** | FKs reales a nivel DB | FKs lógicas (linkedRecord) — Airtable valida pero menos estricto |
| **Transacciones** | ACID completo | Sin transacciones — handler secuencial con log de errores |
| **Polimorfismo** | 1 item_tipo + item_id | 3 linkedRecord separados + serializer |
| **Granularidad Contratos.Tipo** | Libre | Limitado a `venta/reserva/alquiler/boleto` + subtipo en Item_Descripcion |
| **Performance búsqueda** | Índices Postgres + ILIKE | filterByFormula con FIND/LOWER — más lento pero OK para volumen Mica |
| **Costo** | VPS Hetzner | Airtable plan actual |

## Verificación

### Backend producción (`agentes.arnaldoayalaestratega.cloud`)
11/11 endpoints OK: activos, inmuebles-renta, inquilinos, pagos-alquiler, liquidaciones, propietarios, asesores, contratos, personas/buscar, personas/{id}, personas/agregar-rol.

### Frontend producción (`system-ia-agencia.vercel.app/system-ia/dev/crm-v2`)
Deployado inmediatamente tras push (cupo Vercel del proyecto Mica está separado del de Robert, por eso no tuvimos el cuello de botella que afectó a Robert por la tarde).

Marcadores verificados en el HTML servido:
- modalContratoUnificado (1), sg-group (13), data-group="agencia" (1)
- cargarRelacionesCliente (2)
- 64 ocurrencias paleta ámbar, 0 purple Robert

## Impacto para promocionar Mica

Antes de esta sesión, Mica tenía un CRM con paneles vacíos ("próximamente") y una desincronización severa entre Clientes / Leads / Inquilinos / Propietarios. Después:

- Los panels GESTIÓN son operativos, no placeholders
- Una persona es un registro único con roles acumulables — nada de duplicados Pedro-lead + Pedro-inquilino + Pedro-propietario
- El bot WhatsApp Mica sigue mandando mensajes a los leads sin cambios (no se tocó esa capa), pero ahora un lead puede ser convertido a cliente con 1 click manteniendo trazabilidad
- El usuario puede mostrar el CRM a una inmobiliaria real sin "el panel que dice próximamente no hay que mirarlo"
