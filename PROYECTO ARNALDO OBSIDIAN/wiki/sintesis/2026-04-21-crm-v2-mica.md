# CRM v2 Mica — setup completo + decisión Embedded Signup compartido

## Síntesis ejecutiva

El 21 de abril de 2026 se replicó para System IA (proyecto de [[micaela-colmenares]]) todo el trabajo que se hizo antes para [[robert-bazan]] (Lovbot): CRM v2 con skin Claude Design + sidebar por subnichos + schema multi-subnicho en la base de datos. La adaptación respeta la regla crítica de aislamiento de stacks — cada proyecto usa su DB, su provider WhatsApp, su orquestador.

Al cierre: CRM v2 Mica funcionando localmente con datos reales (19 leads de Airtable), backend FastAPI sincronizado, pendiente deploy Vercel por cupo agotado hasta mañana. Además, decisión estratégica de Embedded Signup compartido que cambia el horizonte del bot Mica.

### Schema Airtable ampliado (17 tablas)

La base pasó de 5 tablas pre-existentes (`Clientes`, `Propiedades`, `CLIENTES_ACTIVOS`, `BotSessions`, `Clientes_Agencia`) a **17 tablas** con soporte nativo para los 3 subnichos. Las 12 nuevas se crearon vía Airtable Metadata API:

- Core: `Asesores`, `Propietarios`, `Loteos`, `LotesMapa`, `Contratos`, `Visitas`
- Agencia (alquileres): `InmueblesRenta`, `Inquilinos`, `ContratosAlquiler`, `PagosAlquiler`, `Liquidaciones`
- Config: `ConfigCliente` (define `tipo_subnicho`)

Table IDs documentados en [[inmobiliaria-demo-mica-airtable]]. Todas las nuevas quedan vacías — sirven como schema reserve para cuando Mica quiera activar módulo de alquileres.

¿Cómo replicar para Mica el CRM v2 que se terminó para Robert, respetando que Mica tiene stack distinto (Airtable + Evolution API), sin mezclar datos ni depender de infraestructura de Robert?

## Hallazgos y bugs encontrados

### Bug 1 — HTML apuntaba a Vercel como backend

El subagente `proyecto-mica` inicialmente puso:
```js
const SAAS_API = 'https://agentes.system-ia-agencia.vercel.app';
```

Vercel solo sirve HTML estático, no FastAPI. El `initTenant()` fallaba silenciosamente, por lo que:
- No pedía PIN (saltaba el check)
- No cargaba leads (fetch 404)
- No aplicaba colores del tenant

Fix: apuntar a `agentes.arnaldoayalaestratega.cloud` (Coolify Arnaldo donde corre el worker Mica).

### Bug 2 — Faltaba carpeta `dev/js/`

El HTML de Mica referenciaba `js/_crm-helpers.js`, `js/panel-loteos.js`, etc., pero el directorio `demos/SYSTEM-IA/dev/` no tenía carpeta `js/`. Copiados los 7 archivos JS desde el dev de Robert.

### Bug 3 — `/crm/resumenes` devolvía HTTP 500

El handler del worker Mica llamaba a `_check_pg()` que exige Postgres — pero Mica usa Airtable. Además `db.listar_resumenes()` del stack Airtable no aceptaba los 4 parámetros del handler. Fix: reescribir handler para no bloquear por stack + extender firma del stub.

### Bug 4 — Tenant `mica-demo` en Supabase con base Airtable incorrecta

El campo `airtable_base_id` apuntaba a `appXPpRAfb6GH0xzV` (base de Arnaldo/Maicol) mientras las tablas apuntaban a `appA8QxIhBYYAHw0F` (Mica). URL resultante inválida. Fix vía PATCH Supabase REST API.

### Bug 5 — Colores en Supabase heredados de demo anterior

`color_primario: #0f5c3b` (verde oscuro) y `color_acento: #b6ae9a` (beige) — residuo histórico. Actualizados a ámbar/rojo Mica.

## Decisión estratégica — Embedded Signup compartido

**Contexto**: durante esta sesión se confirmó la aprobación del Embedded Signup de Meta para el Tech Provider de Robert. El número que hoy usa Mica (`+54 9 3765 00-5465` sobre Evolution API) se va a migrar al Meta Graph API oficial.

**Decisión**: Mica + clientes propios de Arnaldo van a onboardear sus números vía el TP de Robert como **tránsito técnico**, hasta que cada uno (Arnaldo primero, Mica después) saquen su propio Tech Provider de Meta.

**Implementación**: en el commit `ec33418` ya se preparó la infraestructura para esto:
- PG `waba_clients` con columnas `agencia_origen` + `meta_user_id`
- Endpoints admin FastAPI para `GET /tenants-por-agencia` + `POST /reclasificar-agencia`
- SOP de migración futura documentada en [[meta-tech-provider-onboarding]]
- Cuando un cliente migre del TP Robert al TP propio: 1 click + cero pérdida de datos

**Transparencia con Robert**: Arnaldo le comunica via WhatsApp que va a onboardear clientes propios vía su app TP, como tránsito técnico. No se menciona a Mica (ella decide cuando quiera).

### Impacto técnico en el bot Mica

Hoy el bot Mica corre sobre Evolution API. Cuando se migre a Meta Graph:
1. Ajustar el worker para usar `workers.shared.wa_provider` con `WHATSAPP_PROVIDER=meta`
2. Apuntar al número WABA onboardeado via Embedded Signup de Robert
3. Evolution API puede seguir activo como fallback durante la transición

**Trabajo pendiente para sesión futura.** Evolution sigue funcionando hoy.
