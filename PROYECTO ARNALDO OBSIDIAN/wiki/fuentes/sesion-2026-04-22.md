# Sesión Claude 2026-04-22

## Resumen

Sesión densa que arrancó con un ticket urgente (CRM Maicol caído por CORS) y derivó en 6 trabajos encadenados:

1. Fix puntual CORS Maicol → backend Arnaldo + commit `a681a5c`.
2. Construcción de **monitoreo Capa 1 unificado** (`guardia_critica.py`) cubriendo las 3 agencias (14 checks activos + 2 preparados Mica).
3. Identificación y fix de un **bug histórico** que hacía que cada subagente pidiera tokens Coolify nuevos en lugar de leer los existentes (causa: comillas simples + pipe `|`).
4. **Migración CRM v1 → v2 unificada** para [[wiki/entidades/lovbot-ai|Robert]] y [[wiki/entidades/system-ia|Mica]] (commits `83d24c8`, `a674c3d`, `ab879fb`).
5. Auditoría de [[wiki/conceptos/postgresql|Postgres Robert]] confirmando que las 3 BDs legacy YA NO existen (la memoria estaba desactualizada).
6. Sincronización y arreglo de ambos paneles admin (Robert + Mica) que tenían bugs en el botón "🔗 CRM" apuntando al v1 viejo.

## Ideas clave

1. **El monitor diario `auditor_runner.py` (8am) y el monitor de Capa 1 `guardia_critica.py` (cada 5 min) son productos distintos** y existían desde 2026-04-11, pero el segundo NO funcionaba por env vars faltantes. Antes de "crear" infra, **listar lo que ya hay**.
2. **Los tokens Coolify llevan pipe `|`** (formato Sanctum). En `.env` requieren comillas DOBLES. Comillas simples o sin comillas → `bash source` lo lee como pipe shell → 401 silencioso → loop de "pedir token nuevo".
3. **Los CRMs v2 son multi-tenant**: 1 archivo HTML sirve a N clientes vía `?tenant=<slug>`. No se duplica el HTML por cliente — se duplica solo la BD/base.
4. **Las URLs viejas de los CRMs legacy se preservan vía rewrite Vercel** (no rompen links pegados en chats, mails, etc). Apuntan al v2.
5. **La memoria del bot Lovbot vive dentro de `lovbot_crm_modelo`** (3 tablas: `bot_sessions`, `resumenes_conversacion`, `notas_bot`). Chatwoot tiene su BD interna separada — la integración es solo via API REST.
6. **Patrón "monitor consolidado"**: 1 ciclo = 1 mensaje Telegram con TODOS los servicios caídos juntos (no spam por servicio).

## Decisiones tomadas

1. **CRM v2 es modelo único** para Robert y Mica. Los archivos `crm.html` y `demo-crm-mvp.html` quedan eliminados del repo.
2. **El monitor `guardia_critica.py` es el sistema oficial de Capa 1** (no se necesita un script local paralelo).
3. **Los tokens Coolify se referencian SIEMPRE con comillas dobles** en cualquier `.env` del repo.
4. **Antes de pedir un token al usuario, agotar lookup en ambos `.env`** (raíz Mission Control + backend monorepo).
5. **El admin de cada agencia apunta al CRM v2 de su agencia** (no cross-agencia).
